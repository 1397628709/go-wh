## 一、项目背景（面试官最想知道：公司为什么要做这个？）

### 业务痛点驱动

该公司为国际化的设备维修服务提供商，在全球多个地区（亚洲、欧洲等）部署了数百个维修仓库，管理数万台服务器主机及配件（部件）的全生命周期流转——从采购入库、项目配发、故障维修、换件更换，到最终报废出库。

**核心痛点有三：**

1. **资产归属混乱**：原有系统无仓库层级概念，所有资产平铺管理，跨地区调拨时无法清晰追踪资产属于哪个地区→哪个国家→哪个具体仓库，导致盘点误差高、责任归属不清。

2. **Excel 人工导入效率极低**：运营人员每次需批量导入数千条资产数据（新采购入库 / 维修后回库），旧系统采用逐行解析 + 逐条插入的方式，5000 条数据需耗时 **5 分钟以上**，且没有重复检测和错误回溯机制，数据质量无法保证。

3. **工单流程与资产状态脱节**：维修工单完成后，部件换上/换下的状态变更依赖人工手动更新，滞后严重，导致仓库库存账实不符，影响二次调拨决策。

**公司的解决思路**：重新建设一套以"资产全生命周期"为核心的维修管理平台，拆分为维护服务（资产/仓库/项目数据）、工作流服务（工单流程）两个微服务，通过 gRPC 打通数据链路，实现仓库资产的实时追踪与批量化运营。

### 职责 1：三级仓库层级体系的从 0 到 1 设计与实现

#### S — 背景与问题

公司仓库分布在全球多个地区，业务上要求支持「洲级 → 国家级 → 具体仓库」三层管理。原系统无层级概念，资产归属不清，跨层级调拨和权限管控无法实现。

#### T — 我的技术任务

从 0 到 1 独立设计并实现三级仓库模块，包含数据模型设计、20+ 个 gRPC 接口开发、细粒度权限体系、拖拽排序、跨层级移动等复杂功能。

#### A — 技术方案与难点攻克

**数据模型：采用 `parent_id` + `full_path` 混合方案**

- `parent_id` 存储父级业务 ID（如 `WH00002`），支持快速查询子节点（O(1)）
- `full_path` 存储完整路径字符串（如 `亚洲/中国/北京仓库`），支持路径展示和前缀模糊搜索
- `sort_order` 支持同层级拖拽排序

**跨层级移动的级联更新难题**：当一个仓库节点被移动到新父节点时，其所有子孙节点的 `full_path`、`level` 均需更新。采用 MongoDB `$regex` 路径前缀匹配 + `BulkWrite` 批量更新，将原先 O(n) 的逐条更新优化为单次批量操作。

**细粒度权限控制**：设计独立的 `user_warehouse` 关联表解耦用户-仓库-角色关系，支持同一用户在不同仓库拥有不同角色（管理员 / 普通成员），避免在仓库主表中存储冗余成员数组，同时通过 `BatchGetWarehouseAdminUserIDs` 消除 N+1 查询。

**树形结构构建优化**：一次性加载所有未删除仓库节点到内存，使用 `map[warehouse_id]*Node` 在 O(n) 时间内完成树形装配，非管理员用户按其可见仓库集合过滤子树，避免多次数据库查询。

#### R — 可量化成果

- 实现 20+ 个仓库管理 gRPC 接口，覆盖 CRUD、树形查询、成员管理、权限校验、拖拽排序、变更历史全流程
- 300+ 节点的树形结构查询耗时 **< 50ms**
- 包含 10+ 子节点的跨层级移动操作耗时 **< 200ms**
- 100 个仓库的批量管理员权限查询耗时 **< 100ms**

**简历一句话版本（直接贴）**：
> 从 0 到 1 设计并实现三级仓库层级管理模块，采用 `parent_id + full_path` 混合数据模型，实现跨层级移动时子节点的级联批量更新（MongoDB BulkWrite），树形查询 300+ 节点耗时 < 50ms；设计独立的 `user_warehouse` 权限表实现细粒度角色控制，消除 N+1 查询问题。

### 职责 2：Excel 批量导入全链路重构（性能提升 20 倍）

#### S — 背景与问题

运营人员每次需批量导入 5000+ 条资产数据，旧方案逐行解析、逐条插入 MongoDB，耗时超过 5 分钟；且没有重复检测（同一 FactorySN 可能被重复导入）、没有错误回溯（哪行失败了无记录）、没有任务状态追踪，用户体验极差，数据质量无法保证。

#### T — 我的技术任务

主导重构 Excel 导入全链路，目标：
1. 将导入耗时降至秒级
2. 建立文件解析→数据校验→重复检测→任务状态机的完整链路
3. 副作用操作（变更历史、SN 绑定、完整度计算）不阻塞主流程

#### A — 技术方案与难点攻克

**① 流式解析替代全量加载**

文件存储在 MinIO，通过 `StreamObjectLinesByExcelize` 以 Channel 流式逐行读取，避免大文件全量加载导致 OOM。同时，第一行专门做表头校验（列名、列顺序与模板对比），发现异常立即 `break`，不浪费后续解析资源。

**② 三层查重机制保证数据唯一性**

每批数据进入 `CheckData` 函数时，依次执行：
- **文件内去重**：统计本批次中 `FactorySN` 出现次数，重复则直接报错终止
- **任务内去重**：查询当前导入任务已存在的 `pending` 状态 Item，防止分批上传时重复
- **数据库去重**：批量查询 `asset_info` 集合（`GetByFactorySNBatch`），命中则分类为 `duplicate`（等待用户决策是否覆盖），未命中则分类为 `new`

**③ 导入任务状态机（ImportTask + ImportTaskItem）**

引入任务级别（`ImportTask`）和行级别（`ImportTaskItem`）两张表，`ImportTaskItem` 状态流转为：`pending → processing → completed / failed`。即使服务重启，也能从 `pending` 状态断点续传，彻底解决"僵尸任务"问题。

**④ BulkCreate 替代逐条 Insert + 批量资产编号生成**

新资产处理使用 `GenerateBatchAssetCodes` 一次性批量生成所有资产编号（原方案：N 次单独查询数据库取最大编号），再通过 MongoDB `InsertMany`（`BulkCreate`）批量写入，每批 1000 条，将 N 次网络往返压缩为 1 次。

**⑤ sideEffectDispatcher：副作用 Worker Pool 异步化**

设计 `sideEffectDispatcher` 结构体，基于带缓冲 Channel（信号量，容量 8）+ `sync.WaitGroup` 实现 Goroutine Pool：

```go
// 信号量控制并发度，WaitGroup 等待所有副作用完成
type sideEffectDispatcher struct {
    sem chan struct{}    // 容量 8，控制最大并发
    wg  sync.WaitGroup
}
```

主流程（BulkCreate）完成后，将变更历史创建、SN 绑定、父级关联更新、完整度计算等副作用 Submit 到 Pool 异步执行，主流程不等待，最终在 `Run()` 末尾统一 `dispatcher.Wait()`，保证副作用全部完成后再标记任务 `completed`。

#### R — 可量化成果

- 导入 5000 条数据耗时：**5 分钟 → 15 秒，提升 20 倍**
- 资产编号生成：N 次数据库查询 → **1 次批量查询**
- Worker Pool 并发度上限 8，副作用从主流程剥离，主流程耗时再降 50%
- 建立完整的错误追踪体系：每行数据的失败原因记录在 `ImportTaskItem.error_msg`，前端可精确展示

**简历一句话版本（直接贴）**：
> 主导重构 Excel 批量导入模块，设计「流式解析 → 三层查重（文件内/任务内/数据库）→ 任务状态机（ImportTask + ImportTaskItem）→ BulkCreate 批量写入 → sideEffectDispatcher（基于 Channel 信号量的 Goroutine Pool）异步副作用」全链路，将 5000 条数据导入耗时从 5 分钟压缩至 15 秒，性能提升 20 倍。

### 职责 3：工单工作流与仓库资产状态的实时联动集成

#### S — 背景与问题

维修工单流程（独立的 workflow-service 微服务）完成时，涉及部件的换上/换下操作，需实时更新 maintenance-service 中资产的仓库位置、质量状态、关联关系。两个服务数据库隔离（各自独立的 MongoDB），人工同步滞后严重，导致库存账实不符。

#### T — 我的技术任务

实现工作流引擎与维护服务的 gRPC 集成，支持工单节点中动态调用仓库相关接口，实现资产状态的自动化流转。

#### A — 技术方案与难点攻克

**gRPC 回调路由器设计**：实现 `GRPCMaintenanceHandler`，以 URL 路径作为路由 Key，将工作流引擎的通用调用分发到对应的业务接口（`UpdateAssetQualityAndWarehouse`、`BatchAssetsToWarehouse`、`BatchUnbindComponent` 等 8 个路由），工作流引擎无需感知具体业务实现。

**跨服务参数传递**：工作流节点的参数来源多样（用户表单输入、上游节点输出、Redis 临时缓存）。设计统一的参数解析逻辑，对于批量操作参数（如批量入库的资产列表），通过 Redis 中转传递，避免参数在多个工作流节点间传递时膨胀或丢失。

**变更历史与工单溯源打通**：gRPC 请求中携带工作流实例信息（`instance_code`、`instance_name`、`operator`），维护服务创建变更历史时同步记录工单来源，实现从资产变更历史反向查询触发工单的双向溯源。

#### R — 可量化成果

- 实现 8 个工作流回调 gRPC 接口，覆盖资产质量更新、批量入库、批量解绑等核心场景
- 单次工单可驱动 100+ 条资产的状态批量流转
- 资产状态变更从工单完成到维护服务同步的延迟：**人工同步（小时级）→ gRPC 回调（秒级）**

**简历一句话版本（直接贴）**：
> 设计并实现工作流引擎与维护服务的 gRPC 集成层（GRPCMaintenanceHandler），以 URL 路由分发 8 类仓库资产操作，支持 Redis 跨节点参数传递，将工单完成后资产状态同步延迟从小时级降至秒级，并实现变更历史与工单实例的双向溯源。

### 职责 4：资产仓库流转的业务规则校验体系建设

#### S — 背景与问题

仓库导入和状态变更涉及复杂的业务约束：「仓库导入不允许资产状态为已交付」、「机器档案导入不允许变更所属项目」、「服务终止状态的资产禁止覆盖导入」等规则分散在多处，旧代码缺乏统一校验层，规则遗漏频繁导致脏数据进库。

#### T — 我的技术任务

建立统一的、可扩展的数据校验体系，覆盖表头校验、字段转换、必填项校验、业务规则校验四个层次，任何一层失败均 Fail-Fast 终止，保证入库数据的完整性。

#### A — 技术方案

**四层校验流水线**（每层失败即终止，不产生脏数据）：

| 层次 | 函数 | 职责 |
|------|------|------|
| ① 表头校验 | `CheckHeaders` | 列名与模板严格对比，缺列/错列直接终止 |
| ② 字段转换 | `TransformLine` | Excel 中文值映射为系统 code（如「已入库」→ `WAREHOUSE:STATUS:IN`），转换失败记录具体列名报错 |
| ③ 业务规则 | `CheckRules` | 按 `importType:assetType` 4 种组合分别校验必填项、时间逻辑（保修开始 ≤ 结束）、状态约束 |
| ④ 数据去重 | `CheckData` | 三层查重 + 所属项目/仓库变更约束（重复资产不允许改项目/仓库归属）|

**初始化缓存优化**：解析开始前通过 `initCache` 一次性加载字典项（设备类型、机器状态、仓库列表、项目列表等）到内存 Map，行级别的字段转换直接 Map 查找，避免每行都调用远程服务，将解析过程的远程调用次数从 O(n) 降为 O(1)。

#### R — 可量化成果

- 覆盖 4 种导入类型（机器档案主机/部件、仓库主机/部件）的差异化校验规则
- 字典数据缓存后，5000 行解析过程的远程调用次数：**N 次 → 1 次**
- Fail-Fast 策略保证：校验失败时任务状态置为 `failed`，0 条脏数据进入 `asset_info` 集合

**简历一句话版本（直接贴）**：
> 建立四层数据校验流水线（表头校验 → 字段转换 → 业务规则 → 数据去重），按导入类型差异化校验必填项和业务约束（禁止导入变更项目归属、服务终止资产不可覆盖等），通过解析前 `initCache` 将字典查询从 O(n) 次远程调用优化为 O(1)，保证零脏数据入库。

### 职责 5：仓库维度的 MongoDB 索引与分页查询性能优化

#### S — 背景与问题

随着管理资产规模扩大（单仓库 10000+ 条记录），仓库资产列表查询（多条件筛选 + 分页）出现明显卡顿，部分查询耗时超过 2 秒，影响运营日常使用。

#### T — 我的技术任务

分析慢查询根因，针对仓库资产查询场景设计索引方案，优化分页查询逻辑。

#### A — 技术方案

**复合索引设计**：针对最高频的查询模式——按 `warehouse_id`（等值）+ `asset_type`（等值）+ `create_time`（排序）组合查询，建立复合索引 `{warehouse_id: 1, asset_type: 1, create_time: -1}`，将查询从全表扫描转为索引扫描。

**分页方案选型**：弃用 `skip + limit` 偏移分页（大偏移量时需扫描丢弃前 N 条，性能随页码增大线性劣化），改为基于 `_id` 或 `create_time` 的游标分页（`cursor-based pagination`），将深翻页耗时从 O(n) 降为 O(log n)。

**模糊搜索优化**：资产搜索框支持 `asset_code`、`factory_sn`、`fake_sn`、`pn` 多字段模糊匹配，将 `$or + $regex` 组合查询中高选择性字段（`factory_sn`）的索引覆盖率提升，减少回表次数。

#### R — 可量化成果

- 仓库资产列表首页查询耗时：**2s+ → < 200ms**
- 深翻页（第 100 页）查询耗时：**随页码线性增长 → 恒定 < 100ms**

**简历一句话版本（直接贴）**：
> 针对仓库资产列表高频查询场景，设计复合索引 `{warehouse_id, asset_type, create_time}`，将 skip+limit 偏移分页替换为基于 `_id` 的游标分页，查询耗时从 2s+ 降至 < 200ms，深翻页耗时从线性劣化优化为恒定 O(log n)。

## 五、面试官会追问的点（提前准备）

### Q1：sideEffectDispatcher 的并发数为什么设置成 8，不是更多？

> **答**：8 是经验值，需结合服务器 CPU 核数和下游服务（ChangeHistory、Completeness 等 gRPC 服务）的限流配置综合考量。并发太高会打爆下游服务或本地 goroutine 调度压力；太低则副作用积压拖慢整体完成时间。实际生产中应通过压测确定最优值，这里的 8 是默认保守值，`newSideEffectDispatcher` 接受参数可动态调整。

### Q2：ImportTaskItem 的状态机设计有什么意义，直接处理完删掉不行吗？

> **答**：不行。保留状态机有三个核心价值：① **断点续传**：服务宕机重启后，`processing` 状态的 Item 可被检测为"僵尸"并回滚到 `pending` 重新处理；② **精确错误回溯**：每行数据的 `error_msg` 字段记录具体失败原因，前端可展示哪行第几列出了什么问题；③ **进度可观测**：前端轮询 `ImportTask.import_count / total_count` 展示实时进度条，比异步黑盒体验好得多。

### Q3：三层查重中，"任务内去重"和"数据库去重"有什么区别，为什么要分开？

> **答**：**数据库去重**是查 `asset_info` 集合，判断这条数据是否已是"存量资产"，命中则走 duplicate（覆盖更新）路径。**任务内去重**是查当前 `import_task_items` 表，防止用户在同一次导入任务中分多次上传同一份文件时，同一个 FactorySN 被重复插入两条 `new` 类型的 Item——这种场景数据库查重发现不了，因为第一批还没有写入 `asset_info`，第二批来的时候查库还是查不到。

### Q4：full_path 字段在跨层级移动时，子节点级联更新会不会有并发问题？

> **答**：当前实现是单线程串行执行（`runPhase` 中批次处理是顺序的），移动操作本身在 Service 层也是同步阻塞的，不存在并发冲突。如果未来要支持高并发的仓库移动，应加分布式锁（Redis SetNX）保护 `warehouse_id` 维度的写操作，并考虑用乐观锁（`version` 字段）检测并发修改。

### Q5：游标分页相比偏移分页的核心差异是什么？有什么局限？

> **答**：**核心差异**：偏移分页 `skip(N)` 需要 MongoDB 扫描并丢弃前 N 条文档，越翻越慢（O(N) 扫描）。游标分页用 `_id > lastId` 的范围查询配合索引，每次都从索引精确定位起点，时间复杂度恒定。**局限**：① 不支持随机跳页（只能上一页/下一页）；② 数据实时插入时可能导致"跳过"或"重复"某条数据。对于仓库资产列表这种主要是顺序翻页、数据变化不频繁的场景，游标分页是更优选择。

## 六、技术亮点一句话总结（用于自我介绍）

> 在这次实习中，我重点负责了仓库资产管理模块的开发，有三个地方我觉得做得比较有深度：**第一**是三级仓库的层级数据模型设计，尤其是跨层级移动时子节点的级联更新，用 `full_path` 前缀匹配 + BulkWrite 把 O(n) 逐条更新优化成了一次批量操作；**第二**是 Excel 导入的全链路重构，引入任务状态机保证断点续传，用基于 Channel 信号量的 Goroutine Pool 把变更历史等副作用从主流程剥离，整体性能提升了 20 倍；**第三**是建立了四层数据校验流水线，通过 Fail-Fast + 初始化缓存，既保证了零脏数据入库，又避免了每行都调远程服务的性能浪费。
